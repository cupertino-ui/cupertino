# Cupertino UI — Automation Brief for Sonnet

Goal: complete the Cupertino UI registry by porting every remaining shadcn/ui primitive into this project and re-skinning each with Apple Human Interface Guidelines (HIG) styling.

## Project context

- Cupertino UI is a shadcn-compatible registry. Consumers run `npx shadcn@latest add <url>` and get a HIG-styled drop-in replacement for the original shadcn component.
- Source of truth lives in `registry/<name>/<name>.tsx`. `components/ui/<name>.tsx` is the synced copy used by the docs site.
- The CLI artifact is generated by `npm run registry:build` into `public/r/<name>.json`.
- Read `AGENTS.md` first — this project pins a custom Next.js build with breaking changes. APIs may differ from your training data; check `node_modules/next/dist/docs/` before writing route or config code.
- Read the existing `registry/button/button.tsx`, `registry/switch/switch.tsx`, and `registry/sheet/sheet.tsx` before starting — they define the house style.

## Components to add

Anything from the shadcn catalog not already present in `registry/`. As of now, present components are: badge, button, card, input, kbd, label, separator, sheet, switch, tabs, tooltip.

Remaining (verify against `npx shadcn@latest add --help` or shadcn registry index): accordion, alert, alert-dialog, avatar, breadcrumb, calendar, checkbox, collapsible, command, context-menu, dialog, drawer, dropdown-menu, hover-card, menubar, navigation-menu, pagination, popover, progress, radio-group, resizable, scroll-area, select, skeleton, slider, sonner, table, textarea, toggle, toggle-group.

Skip components that don't fit the HIG aesthetic (e.g. `chart`, `form` — confirm with maintainer if unsure).

## Per-component workflow

For each component, run this loop. Do not batch — one component at a time so each can be reviewed.

1. **Install the upstream shadcn primitive** into `components/ui/`:
   ```bash
   npx shadcn@latest add <name> --yes --overwrite
   ```
2. **Scaffold the registry source** from the installed file:
   ```bash
   npm run component:init <name>
   ```
   This copies `components/ui/<name>.tsx` → `registry/<name>/<name>.tsx`.
3. **Re-skin** `registry/<name>/<name>.tsx` with HIG styling. Match the patterns already used in `registry/button/button.tsx`:
   - Rounded shapes: pill (`rounded-full`) for interactive controls, `rounded-2xl`/`rounded-3xl` for surfaces, `rounded-xl` for inputs.
   - Sizing: 44pt (`h-11`) minimum tap target; 51×31 for toggles.
   - Press feedback: `active:scale-95` with `transition-all duration-200 ease-out`.
   - Focus rings: `focus-visible:ring-3 focus-visible:ring-ring/50`.
   - Typography: tight tracking, medium/semibold weights, San Francisco–like sizing (13/15/17pt scale via Tailwind `text-sm`/`text-base`/`text-lg`).
   - Colors: lean on existing CSS variables (`primary`, `secondary`, `muted`, `destructive`). Destructive uses Apple system red.
   - Add `data-slot`, `data-variant`, `data-size` attributes when variants exist.
   - Keep the public API identical to shadcn so it remains a drop-in replacement.
4. **Register** the new item in `registry.json`. Use this shape (no `target` field — alias resolution handles Laravel/Next paths):
   ```json
   {
     "name": "<name>",
     "type": "registry:ui",
     "title": "<Title>",
     "description": "Apple HIG <name> — <one-line summary>.",
     "dependencies": ["radix-ui"],
     "registryDependencies": [],
     "files": [
       { "path": "registry/<name>/<name>.tsx", "type": "registry:ui" }
     ]
   }
   ```
   Add `class-variance-authority` to `dependencies` only if the component uses `cva`. Add `registryDependencies` (e.g. `["button"]`) when one Cupertino component imports another.
5. **Sync to `components/ui/`** so the docs site picks it up:
   ```bash
   npm run component:sync <name>
   ```
6. **Add a docs section** in `app/docs/page.tsx` following the existing pattern (one `<ComponentDoc />` block with `<Showcase>` examples covering variants/sizes/states).
7. **Build the registry artifact**:
   ```bash
   npm run registry:build
   ```
8. **Verify**:
   - `npm run lint` passes.
   - `npm run build` succeeds.
   - `public/r/<name>.json` was generated and contains `"type": "registry:ui"` on each file with no explicit `target`.
9. **Commit** with a short, single-line conventional-commit message, e.g. `feat: add HIG dialog component`. No body, no trailers.

## Hard rules

- **Drop-in compatibility**: never change a component's exported names, props, or default behavior. Style only.
- **No hardcoded `target`** in `registry.json` files — it breaks Laravel installs. Use `type: registry:ui` and let aliases resolve.
- **No new dependencies** unless the upstream shadcn version already required them. Prefer `radix-ui` umbrella imports (already used).
- **Source of truth is `registry/`**. Never edit `components/ui/` directly — it will be overwritten by `component:sync`.
- **One component per commit**.
- **Stop on ambiguity**: if a component's HIG translation isn't obvious (e.g. `command`, `calendar`), pause and ask rather than guessing.

## Done definition

- Every targeted component has: `registry/<name>/`, entry in `registry.json`, synced `components/ui/<name>.tsx`, docs block in `app/docs/page.tsx`, and built `public/r/<name>.json`.
- `npm run build` is green.
- README/docs intro lists the new components if it currently enumerates them.
